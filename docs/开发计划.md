# skills++ 开发计划

## 1. 文档目标

本文档基于 `PRD.md` 输出可执行的研发计划，用于指导 skills++ MVP 的设计、开发、联调、测试与发布准备。目标是将 PRD 中的产品需求拆解为明确的研发阶段、模块任务、里程碑和验收标准，确保团队可以按阶段稳定交付。

## 2. 开发目标

### 2.1 MVP 目标
1. 完成 5 个 skill 来源站的聚合、标准化和浏览搜索。
2. 完成 10 个 AI 工具的目录规则预置、自动扫描和手动配置。
3. 完成 skill 的安装、卸载、重装和安装日志展示。
4. 完成本地已安装 skill 的统一管理与状态展示。
5. 建立可扩展的来源适配器、工具目录规则和本地元数据模型，为后续新增来源站、AI 工具和更新能力打基础。

### 2.2 非目标
1. 不在 MVP 中实现账号体系、云同步、评论收藏、在线调试。
2. 不在 MVP 中实现完整版本语义比较，首版“更新”以“手动检查 + 重装最新版本”优先。
3. 不在 MVP 中实现批量安装/批量卸载，仅保留后续扩展接口。

## 3. 开发原则

1. **配置优先**：来源站适配、AI 工具目录规则、安装策略尽量配置化，避免写死在核心流程。
2. **本地可追踪**：所有安装、卸载、扫描、失败日志都需要在本地持久化，便于排障与后续更新能力接入。
3. **跨平台优先**：所有路径解析、权限检测、打开目录、文件操作必须兼容 macOS / Windows / Linux。
4. **安全可控**：外部命令仅作为辅助信息来源，不直接信任其输出；安装目标目录需经过白名单和边界校验。
5. **前后端解耦**：前端聚焦交互与状态管理，Rust 侧承接文件系统、目录扫描、安装执行和本地服务能力。

## 4. 建议研发范围拆解

### 4.1 前端模块
1. 应用框架与导航：一级导航、布局、路由、全局状态基础设施。
2. 发现页：来源聚合列表、搜索、筛选、排序、卡片展示。
3. Skill 详情页：详情信息、兼容工具、安装入口、安装说明。
4. 已安装页：已安装列表、状态展示、目录跳转、重装/卸载操作。
5. 工具与目录页：目录扫描结果、默认目录配置、自定义目录增删改启停。
6. 设置页：来源站启停、默认安装目录、缓存清理、日志查看。
7. 安装弹窗 / 抽屉：工具选择、目录选择、冲突提示、安装预览、日志进度。

### 4.2 Rust / 桌面端模块
1. 目录规则引擎：内置工具规则、路径展开、平台差异处理。
2. 目录扫描服务：目录存在性、可写性、skill 数量、扫描状态汇总。
3. 来源聚合服务：来源适配器、字段标准化、缓存、容错。
4. 安装执行引擎：下载 / 拉取 / 解压 / 拷贝 / 校验 / 覆盖处理。
5. 已安装识别服务：扫描本地目录并与数据库元数据合并，生成 InstalledSkill 视图。
6. 外部命令桥接：封装 `npx skills add <owner/repo>` 的辅助探测能力。
7. 日志与错误系统：结构化日志、任务日志、用户可读错误映射。
8. Tauri 命令层：为前端提供稳定 IPC 接口。

### 4.3 数据与持久化模块
1. 来源站配置表
2. Skill 缓存表
3. AI 工具与目录配置表
4. 已安装 skill 表
5. 安装任务与日志表
6. 应用设置表

## 5. 技术方案落地

### 5.1 项目结构建议
建议采用 pnpm workspace + Tauri Monorepo 结构：

```text
apps/
  desktop/              # React + Tauri 前端
src-tauri/
  src/
    commands/           # Tauri IPC 命令
    services/           # 扫描、安装、来源聚合、日志
    adapters/           # 来源站适配器、工具规则适配
    repositories/       # SQLite 数据访问
    models/             # Rust 数据模型
packages/
  shared/               # TS 类型、常量、schema
  ui/                   # 通用 UI 组件
```

### 5.2 核心模型补充
除 PRD 中已有模型外，建议增加以下实体：

#### InstallTask
```ts
type InstallTask = {
  id: string
  skillId?: string
  skillName: string
  toolName: string
  directoryId: string
  action: "install" | "reinstall" | "uninstall" | "scan"
  status: "pending" | "running" | "success" | "failed" | "cancelled"
  startedAt?: string
  finishedAt?: string
  errorMessage?: string
}
```

#### ToolRule
```ts
type ToolRule = {
  toolName: string
  platform: "macos" | "windows" | "linux" | "all"
  candidatePaths: string[]
  detectionHints?: string[]
  installStrategy: "copy" | "git" | "archive" | "skills_cli"
}
```

### 5.3 接口分层建议
1. **前端 API Hook 层**：统一封装查询、变更、任务状态订阅。
2. **Tauri Command 层**：仅暴露稳定命令接口，不直接暴露内部实现细节。
3. **Service 层**：承载业务流程，如“安装 skill”“扫描目录”“同步缓存”。
4. **Repository 层**：处理 SQLite 读写。
5. **Adapter 层**：隔离来源站和外部 CLI 的结构差异。

## 6. 分阶段开发计划

## 6.1 P0：调研与技术验证（第 1 周）

### 目标
完成目录规则、来源站数据抓取方式和安装链路的技术验证，产出可直接进入开发的规则与约束清单。

### 任务
1. 调研 10 个 AI 工具的默认 skill 目录、平台差异、权限要求和是否支持多目录。
2. 调研 5 个来源站的页面结构、是否可直接请求、字段可得性、频率限制和缓存策略。
3. 验证 `skills` CLI 的安装行为、目录发现能力、输出结构和失败模式。
4. 明确 MVP 的统一 SkillItem 字段映射规则与空字段兜底策略。
5. 明确安装执行的最小兼容策略：git 仓库、zip/tar 资源、skills CLI 兼容安装。

### 产出物
1. 工具目录规则清单
2. 来源站适配规则清单
3. 安装策略矩阵
4. 失败场景与错误码清单

### 退出标准
1. 至少完成 10 个工具中的默认目录规则确认或可行兜底方案。
2. 至少完成 5 个来源站的数据获取方案与字段映射。
3. 完成一条从来源数据到本地安装目录的可跑通样例链路。

## 6.2 P1：基础框架与数据层（第 2-3 周）

### 目标
搭建桌面应用骨架、数据库、Tauri 命令层和基础导航，为后续业务开发提供稳定底座。

### 任务
1. 初始化 Tauri + React + TypeScript + Tailwind + Radix UI + TanStack Query 项目。
2. 建立路由、布局、主题、通知、对话框等基础 UI 能力。
3. 建立 SQLite 表结构、迁移机制和 Repository 层。
4. 建立共享类型、前后端通信 schema 和错误码体系。
5. 建立日志系统与本地日志存储。
6. 输出前端页面骨架：发现、已安装、工具与目录、设置。

### 交付结果
1. 应用可启动并完成本地数据库初始化。
2. 前后端 IPC 可用，具备基础查询与写入能力。
3. 核心页面空态可访问。

## 6.3 P2：工具目录与环境初始化（第 4 周）

### 目标
实现自动扫描、目录状态展示、手动配置与默认目录管理。

### 任务
1. 实现工具目录规则模型与路径展开逻辑。
2. 实现默认目录扫描、读写权限检测、skill 数量统计。
3. 实现工具与目录页的扫描结果展示。
4. 实现手动新增目录、编辑目录、禁用目录、设置默认目录。
5. 实现首次启动引导：扫描 → 展示结果 → 补充自定义目录。
6. 接入 `skills` CLI 探测能力作为目录规则辅助校验。

### 交付结果
1. 用户首次启动可看到自动识别出的目录列表。
2. 用户可新增并保存自定义目录。
3. 系统可推荐默认安装目录。

## 6.4 P3：来源站聚合与发现页（第 5-6 周）

### 目标
完成来源数据抓取、标准化、缓存、搜索与筛选。

### 任务
1. 实现 5 个来源站适配器。
2. 定义统一 SkillItem 转换与兼容工具推断逻辑。
3. 实现本地缓存与刷新策略。
4. 实现发现页列表、搜索、来源筛选、标签筛选、工具筛选、排序。
5. 实现 Skill 详情页信息展示。
6. 实现来源站启用/禁用设置。

### 交付结果
1. 用户可浏览来自 5 个来源站的统一 skill 列表。
2. 搜索和筛选可正常工作。
3. 详情页可展示来源信息、仓库地址和兼容工具。

## 6.5 P4：安装链路与日志（第 7-8 周）

### 目标
完成 skill 安装、覆盖冲突处理、安装预览和安装日志能力。

### 任务
1. 实现安装弹窗：工具选择、目录选择、安装预览。
2. 实现安装执行引擎，支持标准仓库安装和 skills CLI 兼容安装。
3. 实现目录结构校验与安装后校验。
4. 实现同名 skill 冲突处理：覆盖、跳过、另存为。
5. 实现安装任务进度、简洁日志/详细日志切换。
6. 实现失败错误映射与用户提示。

### 交付结果
1. 用户可在 3 步内完成安装。
2. 安装过程可见、可追踪、失败原因明确。
3. 安装结果会写入本地 InstalledSkill 与 InstallTask 元数据。

## 6.6 P5：已安装管理（第 9 周）

### 目标
完成本地已安装 skill 的识别、列表展示和管理操作。

### 任务
1. 实现目录扫描结果与安装元数据融合。
2. 实现已安装列表：名称、工具、目录、来源、安装时间、状态。
3. 实现打开目录、查看来源详情、重装、卸载。
4. 实现“已变更 / 缺失 / 正常”状态识别。
5. 实现手动检查更新的最小链路（如仅比较来源可达性或触发重装）。

### 交付结果
1. 已安装页可准确展示本机 skill 列表。
2. 卸载后对应目录内容与元数据同步移除。
3. 重装流程与安装日志链路打通。

## 6.7 P6：联调、测试与发布准备（第 10 周）

### 目标
完成跨平台回归、异常场景测试、性能优化和首版发布准备。

### 任务
1. 完成单元测试、组件测试、Adapter Mock 测试。
2. 完成安装/卸载/扫描主链路联调。
3. 完成缓存、并发任务、错误提示、权限异常回归。
4. 完成 macOS / Windows / Linux 跨平台验证。
5. 完成打包、签名、自动更新预留和发布说明。

### 交付结果
1. MVP 验收项全部通过。
2. 构建产物可在 3 个平台完成基础安装和运行。

## 11. 主要风险与应对

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 来源站结构变化 | 列表抓取失败或字段缺失 | Adapter 隔离、字段容错、缓存兜底、来源禁用开关 |
| 工具目录规则不稳定 | 扫描不准、安装目录错误 | 规则配置化、CLI 辅助校验、用户手动配置兜底 |
| 外部 CLI 行为变化 | 安装或探测链路不可控 | 不强依赖 CLI，保留本地安装流程和本地规则模型 |
| 跨平台文件系统差异 | 安装失败、权限异常 | 抽象路径与文件系统操作，建立平台专项测试 |
| 来源数据质量不一致 | 兼容工具/更新时间等字段不完整 | 标准化时允许空字段，UI 明确展示“未知” |
| 安装覆盖风险 | 用户已有 skill 被误覆盖 | 安装前预览、冲突确认、默认不静默覆盖 |

## 12. 人员分工建议

### 12.1 最小团队配置
1. 前端开发 1-2 人：页面、组件、状态管理、交互流程。
2. Rust / Tauri 开发 1-2 人：扫描、安装、持久化、命令层。
3. 测试/产品兼任 1 人：规则验证、跨平台回归、验收组织。

### 12.2 并行开发建议
1. 前端与 Rust 通过共享 schema 并行推进。
2. 来源站适配器与工具目录规则可并行调研和开发。
3. 已安装页开发可在安装引擎完成前先接入 mock 数据推进。

## 13. 建议优先确认事项

以下问题建议在 P0 结束前定稿，避免影响后续实现：

1. MVP 的“更新”是否仅定义为“重新安装最新版本”。
2. 来源站缓存策略：是否允许本地缓存、缓存有效期、手动刷新规则。
3. 兼容工具字段的推断规则：仅靠来源标签，还是允许人工规则映射。
4. 同名 skill 的“另存为”是否需要统一命名规范。
5. 是否要求首版就支持批量操作的底层能力预留。

## 14. 最终交付定义

当以下条件全部满足时，视为 MVP 开发完成：

1. 用户可从聚合来源中发现 skill，并在 GUI 中完成安装。
2. 用户可看到自动扫描到的工具目录，并可补充自定义目录。
3. 用户可查看已安装 skill，并执行重装、卸载等操作。
4. 应用在 macOS、Windows、Linux 上均能完成核心链路验证。
5. 主要异常场景具备清晰提示和本地日志支撑。
