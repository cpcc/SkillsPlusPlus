## 目标

在 skills++ 桌面应用中实现 4 种 skill 安装策略（`git` / `copy` / `archive` / `skills_cli`），并与 `main` 分支的 HTTP Bridge 功能合并。

### 4 项关键决策

1. **Adapter 默认 + 用户可覆盖**：每个 source adapter 声明默认策略，InstallDialog 中可改，选中后存表，重装/更新沿用。
2. **完全兼容 vercel-labs lockfile 格式**：复用 `~/.agents/skills/<name>/` 规范存储 + `~/.agents/.skill-lock.json`，与本机已有 `npx skills` 安装互通。
3. **强制校验 SKILL.md**：装完必须有 SKILL.md，文件夹名按 frontmatter 的 `name` 规范化；没有 SKILL.md 视为失败。
4. **本轮一起做非 git 更新检测**：copy/archive/skills_cli 用目录 hash 比对。

### 第二阶段：运行时验证

`pnpm tauri dev` 启动后发现运行时错误：

```
no such column: install_strategy in SELECT ... FROM skill_cache
```

#### 根因分析

旧版 `migrate()` 从未设置 `PRAGMA user_version`，导致：
1. 现有 DB 的 `user_version == 0`（被误判为全新库）
2. `CREATE TABLE IF NOT EXISTS` 对已有表是 no-op
3. 更糟：前一次有 bug 的运行已把 `user_version` 写成 2，但列没加

#### 修复方案

`ensure_v2_columns()` 始终幂等执行，无论 `user_version` 处于何种状态：

```rust
pub fn migrate(conn: &Connection) -> SqliteResult<()> {
    let current = conn.query_row("PRAGMA user_version", ...)?;
    if current == 0 {
        let has_legacy_tables = /* 检查 installed_skills 表是否存在 */;
        if has_legacy_tables == 0 {
            conn.execute_batch(SCHEMA_V2)?;  // 真·全新库
        }
    }
    ensure_v2_columns(conn)?;  // ← 始终执行，幂等
    conn.execute_batch("PRAGMA user_version = 2")?;
    Ok(())
}
```

3 个专门测试覆盖了三种 DB 状态的自愈。

## 关键经验

### 1. DB 迁移不能只依赖 user_version

旧代码从未设置 `user_version`，导致 `version == 0` 既可能是全新库也可能是遗留库。解法：用「核心表是否存在」做二次判断 + 始终跑幂等列补齐。

### 2. 长分支合并时先理解对方的架构模式

main 的 HTTP Bridge 引入了 `_inner` / wrapper 分离模式（让 HTTP bridge 能复用 `_inner` 函数）。合并时不能简单保留某一侧，需要理解模式后统一适配。

### 3. 零依赖解析器足够用

SKILL.md 的 YAML frontmatter 只需要 `name` 和 `description` 两个字段，手写 30 行解析器比引 `serde_yaml` 更轻量。

### 4. Magic bytes 比 URL 后缀更可靠

下载的归档可能 `.tar.gz` 实际是 zip，或反之。用前 4 字节判断格式（gzip `1f 8b`，zip `50 4b 03 04`）比信任 URL 扩展名更健壮。
