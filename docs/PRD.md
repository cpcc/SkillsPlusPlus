# skills++

## 1. 产品概述

### 1.1 产品名称
skills++

### 1.2 产品定位
一个桌面端 AI Skill 安装与管理工具，帮助用户在多个 skill 来源站之间统一浏览、筛选、安装、更新和卸载 skill，并将 skill 安装到不同 AI 工具支持的本地目录中。

### 1.3 背景与问题
当前 AI skill 生态分散在多个站点，安装方式和目录规范不统一。用户通常会遇到以下问题：

1. skill 来源分散，需要逐站点搜索与比较。
2. 不同 AI 工具的 skill 安装目录不同，甚至同一个工具支持多个目录。
3. 安装过程依赖命令行，对普通用户不友好。
4. 已安装 skill 缺少统一管理入口，不便查看来源、版本、状态和安装位置。

本产品旨在提供统一入口，降低 AI skill 的发现、安装和管理成本。

## 2. 产品目标

### 2.1 核心目标
1. 统一聚合多个 skill 来源站点。
2. 自动识别本机 AI 工具及其 skill 安装目录。
3. 提供图形化的一键安装、更新、卸载与重装能力。
4. 让用户清楚知道“某个 skill 从哪里来、安装到了哪里、当前状态如何”。

### 2.2 成功标准
MVP 阶段满足以下标准：

1. 支持至少 5 个 skill 来源站的浏览与搜索。
2. 支持首版预置的 10 个 AI 工具的安装目录识别与安装。
3. 用户可在 3 步内完成 skill 安装。
4. 用户可在应用内查看已安装 skill 列表、安装位置和来源信息。

## 3. 目标用户

### 3.1 核心用户
1. 使用 Codex、Claude、Cursor、GitHub Copilot 等 AI 工具的开发者。
2. 经常尝试新 skill，希望快速安装和切换的高级用户。
3. 不熟悉命令行，但希望使用 skill 生态的普通用户。

### 3.2 用户特征
1. 需要同时使用多个 AI 工具。
2. 对 skill 的来源可信度、安装位置和可维护性敏感。
3. 希望减少手动复制、下载、解压、移动目录等操作。

## 4. 使用场景

### 场景 A：发现并安装 skill
用户打开应用，浏览多个 skill 来源站，搜索想要的 skill，查看详情后选择目标 AI 工具与安装目录，一键完成安装。

### 场景 B：管理已安装 skill
用户查看本地已安装 skill，确认它们分别安装在哪个 AI 工具目录中，并执行更新、重装或卸载。

### 场景 C：多目录安装
某些 AI 工具支持多个 skill 目录。用户希望在安装时明确选择目标目录，或配置默认目录。

### 场景 D：环境初始化
用户首次打开应用时，应用自动扫描本机已安装的 AI 工具和 skill 目录，并引导用户补充未识别的自定义目录。

## 5. 产品范围

### 5.1 MVP 范围
1. 聚合 skill 来源站点：
   - https://skillhub.cn/
   - https://skills.sh/
   - https://skillsmp.com/
   - https://clawhub.ai/skills
   - https://lobehub.com/skills
2. 首版同时支持 macOS、Windows、Linux。
3. 首版预置支持以下 AI 工具：Codex、Claude、Cursor、OpenCode、GitHub Copilot、Antigravity、Gemini CLI、Kimi Code CLI、OpenClaw、CodeBuddy。
4. 同时预留更多 AI 工具接入能力，目录规则与安装适配采用配置化扩展。
5. 支持 skill 浏览、搜索、筛选和详情查看。
6. 支持本机 AI 工具检测与目录识别。
7. 支持 skill 安装、卸载、重装。
8. 支持本地已安装 skill 列表展示。
9. 支持手动配置 skill 目录。

### 5.2 非 MVP 范围
1. skill 内容编辑器。
2. skill 在线运行或调试环境。
3. 账户体系与云端登录。
4. 跨设备同步（可作为后续版本）。
5. 社区评论、评分、收藏系统。

## 6. 功能需求

### 6.1 来源站聚合

#### 6.1.1 来源接入
系统需要支持多个 skill 来源站的数据接入，并对外展示统一结构的数据卡片。

#### 6.1.2 标准化字段
对不同来源站数据进行标准化，至少包含：

1. skill 名称
2. 作者 / 组织
3. 来源站点
4. 仓库地址 / 详情页地址
5. 简介
6. 标签
7. 最近更新时间（若可获取）
8. 安装方式（若可获取）
9. 兼容工具（若可推断）

#### 6.1.3 浏览与筛选
支持以下浏览能力：

1. 全局搜索
2. 按来源站筛选
3. 按 AI 工具兼容性筛选
4. 按标签筛选
5. 按更新时间或热度排序（若来源可提供）

### 6.2 AI 工具与安装目录管理

#### 6.2.0 首版支持工具
首版预置支持以下 AI 工具：

1. Codex
2. Claude
3. Cursor
4. OpenCode
5. GitHub Copilot
6. Antigravity
7. Gemini CLI
8. Kimi Code CLI
9. OpenClaw
10. CodeBuddy

同时，系统架构需预留更多 AI 工具接入能力，新增工具应尽量通过配置扩展完成，而不是修改核心流程。

#### 6.2.1 预置目录规则
应用内置首版支持工具的默认 skill 目录规则。已确认规则可直接内置，未确认规则可通过调研结果、外部命令辅助识别和用户手动配置共同补足。例如：

1. Codex：`~/.codex/skills`、`~/.agents/skills`
2. Claude：`~/.claude/skills`

#### 6.2.2 自动识别
应用启动后需要：

1. 扫描本机已知默认目录是否存在。
2. 标记目录可用性、读写权限和当前 skill 数量。
3. 将扫描结果展示给用户。

#### 6.2.3 外部命令辅助识别
应用可通过 `npx skills add <owner/repo>` 所提供的信息，辅助确认 AI 工具支持的 skill 安装目录与安装方式。

说明：该能力应优先用于“目录发现与规则校验”，而不是直接作为唯一安装路径。应用需保留自身的目录模型和安装流程。

#### 6.2.4 自定义目录
用户可：

1. 手动新增 skill 目录。
2. 指定目录对应的 AI 工具。
3. 设置默认安装目录。
4. 禁用某个目录但保留配置。

### 6.3 Skill 安装

#### 6.3.1 安装流程
用户在 skill 详情页点击“安装”后，系统应提供：

1. 目标 AI 工具选择
2. 目标目录选择
3. 安装前预览（来源、目标位置、可能覆盖的内容）
4. 安装进度与日志

#### 6.3.2 安装方式
MVP 优先支持以下方式：

1. 基于仓库地址或公开资源的标准安装流程。
2. 与 `skills` CLI 兼容的安装流程。
3. 安装后本地目录结构校验。

#### 6.3.3 冲突处理
若目标目录已存在同名 skill，系统需提示用户选择：

1. 覆盖安装
2. 跳过
3. 另存为新名称（若安装方式允许）

### 6.4 已安装 Skill 管理

#### 6.4.1 列表展示
已安装列表至少显示：

1. skill 名称
2. 所属 AI 工具
3. 安装目录
4. 来源站 / 来源仓库
5. 安装时间
6. 当前状态（正常 / 缺失 / 已变更 / 待更新）

#### 6.4.2 操作能力
支持以下操作：

1. 打开安装目录
2. 查看来源详情
3. 重装
4. 卸载
5. 检查更新（MVP 可先提供手动检查）

### 6.5 设置中心
设置页至少包含：

1. AI 工具目录配置
2. 默认安装目录
3. 来源站启用/禁用
4. 缓存清理
5. 日志查看

### 6.6 错误提示与可观测性
安装失败时需要明确提示失败原因，例如：

1. 目录不存在
2. 无写权限
3. 网络错误
4. 来源数据缺失
5. 安装脚本不兼容

同时，应用需保留本地日志，便于排查。

## 7. 非功能需求

### 7.1 性能
1. 主界面首次加载时间 <= 3 秒（已有缓存前提下）。
2. 搜索与筛选操作应在 300ms 内反馈。
3. 单个 skill 安装状态应实时更新，不阻塞主界面。

### 7.2 易用性
1. 安装路径选择应对新手友好，默认推荐可用目录。
2. 安装日志需要“简洁模式 / 详细模式”两种展示层级。
3. 首次启动应提供目录扫描与初始化引导。

### 7.3 安全性
1. 明确展示 skill 来源，避免用户误装未知来源内容。
2. 对安装目标目录进行白名单约束，避免误写系统关键目录。
3. 对外部命令执行结果进行边界校验，不直接信任第三方输出。

### 7.4 可扩展性
1. 来源站接入应采用适配器模式，便于后续新增站点。
2. AI 工具目录规则应采用配置化模型，便于后续扩展新工具。
3. 安装记录与元数据应本地持久化，支持未来更新检查与同步能力。
4. 目录扫描、路径解析和文件操作需具备跨平台兼容性，覆盖 macOS、Windows、Linux 的路径差异与权限差异。

## 8. 信息架构

建议采用以下一级导航：

1. 发现：浏览所有来源站的 skill
2. 已安装：查看和管理本地 skill
3. 工具与目录：管理 AI 工具及安装路径
4. 设置：来源、缓存、日志、实验功能

## 9. 核心页面

### 9.1 发现页
包含搜索框、来源筛选、标签筛选、skill 卡片列表。

### 9.2 Skill 详情页
展示 skill 介绍、来源信息、兼容工具、仓库地址、安装按钮、安装说明。

### 9.3 已安装页
展示本机已安装 skill 列表，并支持筛选、状态查看和管理操作。

### 9.4 工具与目录页
展示 AI 工具、其关联目录、状态检测结果，支持手动编辑。

### 9.5 设置页
展示应用偏好、日志、缓存和实验特性配置。

```ts
  id: string
```

```ts
type AiToolDirectory = {
  id: string
  toolName: string
  path: string
  isDefault: boolean
  isDetected: boolean

  enabled: boolean
```

## 11. 技术方案建议

建议采用以下技术栈：

| 层级 | 技术 | 用途 |

| 前端 | React 18 + TypeScript 5 | 组件化 UI、严格类型 |

## 12. 里程碑规划

### P0：调研与建模
1. 调研 `vercel-labs/skills` 及相关 CLI 行为。
2. 梳理各 AI 工具的 skill 安装目录规则。
3. 调研 5 个来源站的抓取方式、字段结构、反爬限制和可用性。
4. 输出目录规则与来源适配方案。

### P1：MVP 开发
1. 完成桌面壳、基础导航和本地数据库。
2. 完成来源站聚合、搜索与详情展示。
3. 完成目录扫描、手动目录配置。
4. 完成安装、卸载、重装流程。
5. 完成已安装列表。

### P2：增强能力
1. 增加更新检查与版本比较。
2. 增加来源站启用/禁用。
3. 增加错误日志与诊断页。
4. 优化安装冲突处理与回滚机制。

### P3：高级能力
1. WebDAV 同步配置。
2. 更多 AI 工具支持。
3. 更完善的更新检测与批量操作。

## 13. MVP 验收标准

1. 用户首次启动后可看到自动扫描到的 AI 工具目录。
2. 用户可从任一来源站浏览并搜索 skill。
3. 用户可在图形界面内完成 skill 安装，无需手动移动文件。
4. 安装完成后，已安装页可展示该 skill 的工具、目录与来源。
5. 卸载后，本地目标目录中的对应 skill 可被正确移除。
6. 关键失败场景均有明确报错提示。

## 14. 风险与注意事项

1. 不同来源站的数据结构可能不稳定，需通过适配器与容错策略隔离风险。
2. 不同 AI 工具的 skill 目录规范可能变化，需保持规则可更新。
3. 部分 skill 安装流程可能依赖第三方命令，需做好失败回显与回退策略。
4. 若使用第三方 CLI 获取目录信息，需避免强耦合，保留本地规则兜底。

## 15. 待确认问题

1. “更新”能力是否需要精确到版本号，还是首版仅支持重装最新版本。
2. 来源站数据是否允许缓存以及缓存刷新频率。
3. 是否需要批量安装 / 批量卸载能力。
